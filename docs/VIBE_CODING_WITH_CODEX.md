## 1. 一句话结论

生产级 Vibe Coding 不是：

> 描述一个想法 → 接受 AI 的所有代码 → 看到能运行就结束。

更可靠的范式是：

> 人定义意图、边界和验收标准；Codex 探索、计划、实现、运行工具和生成证据；仓库通过指引、测试、CI、权限和评审把正确做法变成可重复的系统。

可以把它概括为：

```text
Specify → Explore → Plan → Implement → Verify → Review → Integrate → Learn
定义       探索       计划      实现        验证       评审      集成        沉淀
```

真正重要的不是一次提示词有多漂亮，而是失败以后，能否把缺失的上下文、测试、工具或护栏写回仓库，使同类错误以后更难再次发生。

## 2. Vibe Coding 到底是什么

### 2.1 三种成熟度

| 层级 | 工作方式 | 适用场景 | 主要风险 |
| --- | --- | --- | --- |
| 纯 Vibe Coding | 只描述效果，快速接受生成结果，很少看 diff | 一次性原型、探索想法 | 隐藏 Bug、安全问题、结构迅速腐化 |
| AI 辅助开发 | 人设计并逐段让 AI 编写，人工检查较多 | 日常功能和重构 | 人工成为验证瓶颈，规则仍可能散落在聊天里 |
| Verified Agentic Engineering | 仓库给代理地图、工具、测试和边界；代理端到端执行并提供证据 | 长期维护、生产软件 | 前期要投资测试、文档和自动化护栏 |

### 2.2 为什么“能运行”不等于“完成”

软件可能在演示路径上正常，却仍然存在：

- 首次启动、刷新、跳转、取消、超时等生命周期错误；
- 对某个例子有效、对同类输入无效的硬编码；
- 并发响应覆盖新状态的竞态；
- 只在开发者机器通过的环境依赖；
- 密钥进入日志、同步存储或测试夹具；
- 不必要的兼容代码和重复状态机；
- 没有失败测试，所以后来又被“优化”回去。

因此，完成标准应从“我看见它工作”升级为“验收标准得到可重复证据支持”。

## 3. 当前最佳范式：Verified Agentic Engineering

OpenAI 的 Harness Engineering 实践强调：当代理反复失败时，不是简单要求它“更努力”，而是识别缺少的能力——上下文、工具、抽象、验证或护栏——并让这些能力变得可读、可执行、可检查。参见 [Harness engineering](https://openai.com/index/harness-engineering/)。

### 3.1 八条核心原则

1. **先定义成功，再写代码**
   必须有可观察的预期结果、非目标和验证方式。

2. **先探索，再计划，再实现**
   多文件、高风险或陌生模块先进入 Plan mode；不要让代理在没有理解状态流时直接改代码。

3. **修复不变量，不修复例子**
   时间戳、句子、视频 ID 和某次日志只是证据，不应直接成为生产条件。

4. **仓库是长期记忆，聊天不是**
   稳定规则写进 `AGENTS.md`、架构文档、测试、脚本和 CI。

5. **验证与实现同等重要**
   Codex 不只负责生成代码，也要复现、运行检查、审查 diff 并汇报残余风险。

6. **小任务、小 diff、小检查点**
   每个变更围绕一个可描述的不变量；大目标拆成可以单独验证的阶段。

7. **边界处验证所有外部数据**
   网络响应、模型输出、用户 JSON、文件和连接器内容都不是可信结构。

8. **权限最小化，危险动作显式化**
   日常读写在受控工作区内自动完成；网络、工作区外写入、删除、发布和外部消息需要更清晰的授权边界。

### 3.2 最佳实践不是“更多文档”

文档应该是一张地图，而不是百科全书：

- 根目录只写全局原则、架构入口、验证命令；
- 子目录写该模块独有的不变量；
- 详细流程放到专题文档；
- 能机械检查的规则交给脚本、测试、lint 或 CI；
- 每条新规则最好来源于真实、重复出现的摩擦。

## 4. 人与 Codex 的职责边界

### 4.1 人最应该负责什么

- 决定要解决哪个问题以及为什么值得解决；
- 定义产品语义、权衡和不能破坏的行为；
- 对高风险权限、数据、兼容性和发布做最终判断；
- 识别“结果虽然通过测试，但方向不对”的情况；
- 提供真实使用反馈和无法从代码推断的业务知识；
- 决定何时建立稳定检查点、发布或回滚。

### 4.2 Codex 最适合负责什么

- 搜索代码、追踪调用和状态流；
- 从日志构造最小复现；
- 提出方案和权衡；
- 修改多个相关文件；
- 编写单元、集成、随机不变量和回归测试；
- 运行命令、读取错误并迭代；
- 审查 diff、查找类似问题和兼容残留；
- 更新文档、CI、迁移脚本和开发工具；
- 在获得明确授权时操作外部系统。

### 4.3 不应外包给“感觉”的事项

- 删除数据、覆盖远端、强制推送；
- 扩大浏览器或云权限；
- 发送邮件、消息、创建公开 PR 或发布版本；
- 选择会改变产品语义的默认行为；
- 接受无法复现、无法解释、无法验证的安全修复；
- 把“测试通过”自动等同于“用户体验正确”。

## 5. 标准开发闭环

### 5.1 第 0 步：建立干净基线

开始任务前确认：

```bash
git status --short
git log -5 --oneline
npm run check
```

目标：

- 知道工作区中哪些改动属于当前任务；
- 知道基线是否已经失败；
- 避免把用户未提交的工作覆盖掉；
- 大任务开始前创建分支或 worktree；
- 不要在一个长期脏工作区中无限叠加无关功能。

### 5.2 第 1 步：定义问题

至少写清楚：

- 当前可观察行为；
- 期望行为；
- 一个具体例子；
- 必须保持不变的东西；
- 已知日志、截图、错误或时间点；
- 成功后如何验证。

坏请求：

```text
修复所有字幕问题。
```

更好的请求：

```text
问题：首次打开浏览器后的第一个 YouTube 视频能显示原文，但译文一直为空。
期望：首次会话、首次视频也应显示加载状态，并在请求完成后显示译文。
必须保持：后续视频切换和 seek 的取消语义不变；不要增加供应商特判。
证据：附上从 content-boot 到 batch-complete 的最小日志区间。
验证：新增首次启动状态机回归测试，运行 npm run check，并执行 fresh browser smoke test。
```

### 5.3 第 2 步：探索

要求 Codex：

1. 先读 `AGENTS.md` 和架构文档；
2. 定位数据从输入到显示的完整路径；
3. 标出状态所有者、缓存、并发和取消点；
4. 解释证据支持什么、不能支持什么；
5. 列出候选根因和区分它们的检测方法；
6. 此阶段不要修改代码。

推荐提示：

```text
先不要修改。请追踪这段日志对应的完整状态流，列出最可能的三个根因，
为每个根因给出支持/反对证据，以及一个能区分它们的最小测试。
```

### 5.4 第 3 步：计划

好的计划不是文件清单，而是因果链：

```text
失败不变量 → 修复所在边界 → 状态迁移变化 → 回归测试 → 集成验证 → 风险
```

计划应包含：

- 根因假设；
- 修改的状态所有者；
- 不会采用的方案及原因；
- 测试如何在旧实现失败；
- 是否涉及迁移、权限、兼容或数据格式；
- 手动验证清单；
- 回滚点。

### 5.5 第 4 步：实现

实施规则：

- 先写最小失败测试或复现夹具；
- 改负责该不变量的上游模块；
- 不要在展示层掩盖数据层错误；
- 不要复制另一个状态变量“暂时修好”；
- 保留真实错误原因，不要统一吞成“失败”；
- 每个有意义的编辑后跑最近测试；
- 发现方案不成立就退回计划，不要继续叠补丁。

### 5.6 第 5 步：验证

验证分层：

| 层 | 证明什么 | 示例 |
| --- | --- | --- |
| 纯函数测试 | 输入输出、边界、结构校验 | usage 归一化、分页计划 |
| 状态机测试 | 顺序、取消、旧响应、重入 | seek、视频切换、配置 revision |
| 传输测试 | 任意字节切块、超时、错误保真 | SSE、JSONL、usage-only event |
| 集成测试 | 多模块契约 | popup 配置落地到 background 请求 |
| 浏览器 Smoke/E2E | Chrome 生命周期和真实 DOM | 首次视频、全屏、扩展重载 |
| 人工验收 | 产品语义和视觉质量 | 字幕是否自然、UI 是否好用 |

验证命令必须记录退出码和结论，而不是只写“已测试”。

### 5.7 第 6 步：Review

Review 至少回答：

- 代码是否真正满足用户意图；
- 是否有旧响应覆盖新状态；
- 是否出现 phrase/video/timestamp 特判；
- 是否增加重复状态所有者；
- 是否添加不必要的新旧协议兼容；
- 是否扩大权限或泄露密钥；
- 测试是否只在证明实现细节，而没有证明行为；
- 文档是否与真实代码一致；
- diff 是否混入无关修改。

### 5.8 第 7 步：集成和沉淀

- 创建清晰、单一目的的提交；
- 保留失败复现和测试；
- 把重复提示词沉淀为 `AGENTS.md`、Skill 或脚本；
- 把稳定检查放进 CI；
- 把反复出现的错误抽象成架构不变量；
- 记录未完成的风险，不把它藏在“完成”后面。

## 6. 如何写出可执行的任务

### 6.1 推荐结构：Issue 风格

```text
标题：<一个结果，不是一个实现动作>

背景：为什么现在要改。
当前行为：用户实际看到什么。
期望行为：可观察的验收标准。
证据：日志、截图、文件、测试或链接。
范围：涉及的模块/文件（如果知道）。
必须保持：不能破坏的行为。
非目标：明确不在本次解决的事项。
验证：测试命令、手动路径、预期输出。
交付：代码、测试、文档、迁移或报告。
```

### 6.2 给 Codex 足够信息，但不要预设错误答案

推荐：

```text
“46:54 附近译文等待 30 秒，日志中第一次请求 firstByte=145ms，随后正文停住。
请区分网络连接、服务端生成、客户端流解析和提交窗口阻塞，不要先假设是 DeepSeek 慢。”
```

不推荐：

```text
“肯定是 DeepSeek 并发问题，给它加三个重试。”
```

### 6.3 明确授权边界

例子：

```text
可以修改代码、测试和本地文档并运行检查；不要提交、推送、发布、扩大 manifest 权限，
也不要向外部服务发送任何真实 API Key。
```

### 6.4 明确“只诊断”与“直接修复”

- 只问“为什么”：要求只读调查和证据，不授权修改。
- “诊断并修复”：允许在范围内改代码、测试和文档。
- “评估方案”：要求权衡和建议，不默认实施。
- “彻底修复”：仍应给出明确的不变量和验收标准，不能理解成无限扩大范围。

### 7.2 Plan mode 快速切换

Codex 官方最佳实践记录了两种方式：

- 输入 `/plan`；
- 使用 `Shift + Tab` 切换 Plan mode（若当前桌面/CLI 版本支持该绑定）。

如果本机行为不同，使用 `Ctrl/Cmd + Shift + /` 检查当前快捷键，或者直接使用 `/plan`。

### 7.3 桌面端斜杠命令

在输入框中输入 `/` 可打开当前环境实际支持的命令列表。以下是官方当前列出的主要命令：

| 命令 | 用途 | 推荐时机 |
| --- | --- | --- |
| `/plan` | 切换多步骤规划模式 | 多文件、架构、迁移、复杂 Bug |
| `/goal` | 设置持续目标 | 长时间任务；通常先 `/plan` 再 `/goal` |
| `/review` | 审查未提交改动或与基线分支比较 | 实现和测试完成后 |
| `/status` | 查看任务 ID、上下文使用和速率限制 | 长任务、怀疑上下文不足时 |
| `/model` | 选择当前模型 | 切换速度/深度角色 |
| `/reasoning` | 选择推理强度 | 复杂调试提高，简单机械任务降低 |
| `/fast` | 切换可用的 Fast 服务档 | 小改动和快速迭代 |
| `/compact` | 压缩当前上下文 | 长聊天接近上下文上限时 |
| `/fork` | 把本地聊天复制成新聊天或 worktree | 比较两个方案，避免破坏主线 |
| `/worktree` | 在新的 Git worktree 中运行 | 并行、隔离的大改动 |
| `/side` | 开启临时侧聊，不打断主任务 | 问局部问题或解释概念 |
| `/init` | 为当前项目生成 `AGENTS.md` 初稿 | 新仓库初始化；生成后必须人工精简 |
| `/mcp` | 查看 MCP 连接状态 | 外部工具/数据不可用时 |
| `/ide-context` | 打开或关闭共享 IDE 上下文 | 需要使用当前选择和打开文件时 |
| `/local` | 在选定本地项目运行 | 从云/其他上下文切回本机 |
| `/cloud` | 在云环境运行（可用时） | 长任务、可复现构建、离线委托 |
| `/cloud-environment` | 选择云环境 | 多套环境或依赖时 |
| `/project` | 选择新聊天所属项目 | 切换代码库 |
| `/task` | 创建不绑定项目的聊天 | 通用问题或一次性工作 |
| `/feedback` | 提交反馈，可附带日志 | Codex 产品问题，不是项目 Bug |
| `/approve` | 对最近一次自动审查拒绝批准一次重试 | 理解风险且只需精确重试时 |
| `/memories` | 配置记忆使用/生成 | 管理跨聊天个人偏好 |
| `/personality` | 调整表达风格 | 改沟通方式，不替代工程指令 |
| `/pet` | 显示或隐藏桌面宠物 | 纯界面偏好 |

启用的 Skills 也可输入 `$` 显式选择；自定义 prompts 会显示为 `/prompts:<name>`。

### 7.4 Steer 与 Queue

Codex 正在运行时有两类跟进：

- **Steer**：把消息注入当前运行，用于纠正方向、补充约束、告知新证据；
- **Queue**：保存为下一轮，用于当前工作完成后的跟进。

桌面端可在 **Settings > General > Follow-up behavior** 选择默认行为；界面会显示执行另一个行为的快捷键。因为该按键可以配置和变化，不要在团队文档中硬编码，以本机设置显示为准。

选择原则：

| 情况 | 使用 |
| --- | --- |
| “不要改 manifest，我刚看到你准备扩大权限” | Steer |
| “完成后再顺便更新 README” | Queue |
| “新日志证明不是连接超时，是正文流停住” | Steer |
| “测试通过后再做一次 review” | Queue |

## 9. Plan、Goal、Default/Code、Review 如何配合

### 9.1 Plan mode

适合：

- 需求模糊；
- 多文件或跨执行世界；
- 有迁移、数据、安全或权限风险；
- 需要比较多种架构；
- 用户希望先确认方向再修改。

Plan mode 的产物应是可以检查的执行方案，而不是泛泛待办列表。

### 9.2 Default/Code 工作模式

适合：

- 计划已确定；
- 任务范围清晰；
- 允许修改代码并运行验证；
- Codex 可以自主完成正常实现步骤。

提示中要说明：允许什么、不要做什么、最终验证是什么。

### 9.3 Goal mode

适合需要持续推进、可能跨多轮的明确目标。官方建议通常先 `/plan` 把目标定义清楚，再 `/goal` 固化目标。

Goal 不适合：

- 只有一个简单问题；
- 目标仍然含糊；
- 需要每个决策都实时人工批准；
- 用“不要停止”掩盖缺失的验收标准。

### 9.4 Review mode

实现后进入 `/review`，选择：

- 对比基线分支；
- 审查未提交变更；
- 审查某个提交；
- 使用自定义 review 指令。

Review 应由新的观察角度完成。提示示例：

```text
审查当前未提交改动。优先寻找正确性回归、竞态、首次启动路径、密钥泄露、
不必要兼容和缺失测试。只报告可操作问题，附文件与最小证据；不要修改代码。
```

## 10. 上下文工程：把信息放在正确的位置

| 信息 | 最合适的位置 | 原因 |
| --- | --- | --- |
| 本次任务特殊约束 | 当前提示/聊天 | 一次性，不污染后续任务 |
| 仓库长期规则 | `AGENTS.md` | 自动加载、可版本控制 |
| 子模块不变量 | 子目录 `AGENTS.md` | 越接近代码优先级越高 |
| 详细架构 | `ARCHITECTURE.md`/docs | 供人和代理按需读取 |
| 个人跨项目偏好 | 全局配置/个人指引 | 不应提交到项目 |
| 模型、沙箱、MCP 默认值 | `.codex/config.toml` 或全局 config | 配置而不是自然语言规则 |
| 重复工作流 | Skill | 可带脚本、模板和引用资料 |
| 一组可安装能力 | Plugin | Skills、MCP、Hooks、Apps 的组合 |
| 外部实时数据/动作 | MCP 或 App connector | 不依赖模型记忆和复制粘贴 |
| 强制阻止危险动作 | Hook/CI/权限策略 | 机械执行，不靠模型“记得” |
| 定时检查/提醒 | Automation | 与代码指令分离 |
| 具体可观察行为 | 测试 | 最强、最不含糊的长期规范 |

### 10.1 上下文最小充分原则

- 给入口文件、符号、日志片段和预期，不要无差别粘贴整个仓库；
- 使用文件链接、`@`、截图或明确路径；
- 日志只保留问题发生前后的最小事件窗口；
- 先让 Codex 搜索再决定需要读取哪些文件；
- 长聊天用 `/compact`，新问题用新聊天；
- 旧讨论只有在仍影响当前任务时才应保留。

## 11. AGENTS.md 的正确写法

官方 Codex 最佳实践把 `AGENTS.md` 定义为面向代理的开放格式仓库说明；CLI 可用 `/init` 生成初稿。更接近当前目录的 `AGENTS.md` 会覆盖/细化上层规则。参见 [Codex best practices](https://learn.chatgpt.com/guides/best-practices) 与 [AGENTS.md](https://learn.chatgpt.com/docs/agent-configuration/agents-md)。

### 11.1 根目录应该写

- 项目目标和一张架构地图；
- 构建、测试和完整验证命令；
- 与常见项目不同的约定；
- 数据/安全/兼容边界；
- 变更的 Definition of Done；
- 指向详细文档和子目录规则的链接。

### 11.2 子目录应该写

- 该模块的状态所有权；
- 不能破坏的不变量；
- 典型测试位置；
- 外部输入边界；
- 该目录特有的禁止模式。

### 11.3 不要写

- 从代码一眼能看出的通用事实；
- 空泛的“写高质量代码”；
- 几百条同等优先级规则；
- 已经过期的文件路径；
- 无法检查、彼此矛盾的要求；
- 只为某一次任务存在的临时条件。

### 11.4 推荐骨架

```markdown
# Project agent map

## Architecture
- <入口和边界>

## Required workflow
1. <基线>
2. <复现>
3. <实现>
4. <验证>

## Invariants
- <真实业务/状态不变量>

## Verification
- Focused: `<command>`
- Full: `<command>`

## Safety
- <密钥、权限、外部写入边界>
```

## 12. Skills、Plugins、MCP、Hooks 和 Automations

### 12.1 Skill

用于可重复、任务型流程，例如：

- 读取一类日志并生成最小复现；
- 构建 Chrome 扩展并执行 smoke test；
- 生成并视觉检查 DOCX/PDF；
- 按团队模板做安全 review。

当同一段复杂提示被重复三次以上，考虑把它做成 Skill。

### 12.2 Plugin

当能力需要一起安装和分发时使用 Plugin，例如同时包含：

- Skills；
- MCP server 配置；
- Hooks；
- Apps/连接器；
- 命令和资源模板。

不要为了一个短说明创建 Plugin；那更适合 `AGENTS.md` 或 Skill。

### 12.3 MCP / App connector

用于读取或操作实时外部系统：GitHub、Slack、Notion、数据库、内部 API。优点是：

- 数据是当前的；
- 可以做结构化读取；
- 权限更明确；
- 不需要把私人内容复制到聊天。

外部写操作仍需明确授权；读取能力不自动意味着可以创建、删除或发布。

### 12.4 Hook

适合强制执行生命周期规则：

- 阻止提交密钥；
- 修改后自动格式化；
- 危险命令前强制检查；
- 文件变更后运行针对性测试。

原则：能由确定性工具检查的，不要只写一句自然语言提醒。

### 12.5 Automation

适合稳定、周期性任务：

- 每日依赖/CI 健康检查；
- 定时整理 Bug；
- 发布后监控；
- 提醒处理长期未关闭事项。

还不稳定、需要频繁人工判断的流程不要过早自动化。

## 13. Bug 修复：从案例特判升级为不变量修复

### 13.1 五问法

1. 用户真正观察到什么？
2. 第一个错误状态在哪里出现？
3. 哪个模块拥有这个状态？
4. 哪条一般性不变量被违反？
5. 什么测试能证明它对一类输入成立？

### 13.2 例子：字幕短语被切开

错误做法：

```text
如果文本包含 "friday night" 就不切。
```

正确方向：

- 判断是模型语义边界、请求窗口、时间坐标还是展示分页造成；
- 用字符/词法坐标表达不可丢失的连续覆盖；
- 保证跨 cue 的语义单元仍可被模型引用；
- 展示分页不得反向修改语义单元；
- 用不同短语、随机边界和重叠 cue 验证同一不变量。

### 13.3 例子：翻译很慢

不要只看总耗时。拆成：

```text
调度等待
→ 建连/响应头 first byte
→ 正文生成
→ SSE/JSONL 解析
→ 覆盖提交
→ UI 绘制
```

每一层需要独立指标。`Failed to fetch` 不能自动等于“模型慢”，HTTP 200 也不能证明正文和提交及时。

### 13.4 先失败再修复

高质量回归测试满足：

- 在修复前确实失败；
- 不依赖真实 API 和不稳定公网；
- 输入尽量小；
- 断言用户可观察行为或核心不变量；
- 同类错误再次出现时能清晰指出原因。

## 14. 测试、评审和验证证据

### 14.1 测试策略

- 多数逻辑放在纯函数/状态机测试；
- 网络协议使用人为切块和错误注入；
- 并发使用可控 promise、epoch 和 callback 顺序；
- 时间轴使用随机/属性测试验证单调、无洞、无重复；
- 浏览器生命周期至少有 smoke checklist，条件允许时自动化；
- UI 改动需要截图、DOM 或真实交互证据，而不仅是 CSS 语法通过。

### 14.2 测试反模式

- 为让新实现通过而降低旧断言，却没有更新业务不变量；
- 只匹配源码字符串，不执行行为；
- 只测成功路径；
- 用真实第三方 API 做全部回归，导致慢且不稳定；
- 在测试里复制实现算法；
- 把一个案例句子通过写死预期当成“通用分句测试”；
- 测试通过后不运行完整检查。

### 14.3 Review 输出格式

问题应包含：

```text
严重度：P0/P1/P2/P3
位置：文件与最小行范围
现象：什么输入下会发生
原因：为什么当前实现违反不变量
后果：用户或系统会看到什么
建议：最小修复方向
```

不要报告：

- 没有实际影响的风格偏好；
- 无法复现或没有代码证据的猜测；
- 已被现有测试和保护逻辑明确处理的情况；
- 与当前 diff 无关的整个仓库重写建议。

## 15. Git、提交、分支与 Worktree

### 15.1 一个提交代表一个可解释结果

提交应当：

- 有单一目的；
- 包含对应测试；
- 可以独立 review；
- 在该提交上检查通过；
- 不混入格式化整个仓库或无关清理。

### 15.2 什么时候使用新分支/worktree

- 改动跨多个模块；
- 当前工作区已有用户修改；
- 要比较两个互斥方案；
- 需要并行代理；
- 重构时间较长但主线仍需修 Bug。

桌面端可用 `/worktree` 或 `/fork`；CLI 也支持 `codex fork`。Worktree 的价值是隔离，不是让更多代理同时修改同一批文件。

### 15.3 不安全的 Git 行为

- 未经确认执行 `reset --hard`、覆盖 checkout；
- 对用户改动使用强制恢复；
- 把 `.env`、API Key 或日志隐私提交；
- 在未 review 完整 diff 前推送；
- 为了“干净”删除无法识别的未跟踪文件；
- 在一个数千行 diff 中同时做功能、重构和协议迁移。

## 16. 多代理与并行工作的使用边界

### 16.1 适合并行

- 相互独立的只读调查；
- 一个代理查网络，一个查 UI，一个查测试缺口；
- 多个 review 视角；
- 不同 worktree 的互斥原型；
- 大量独立文件的机械审计。

### 16.2 不适合并行

- 多个代理同时编辑相同文件；
- 根因还没确定就分别“随便修”；
- 强顺序依赖的状态机改造；
- 子任务很小，协调成本超过执行成本；
- 需要共享大量隐含上下文。

### 16.3 推荐多代理提示

```text
请并行做三个只读调查，不要修改：
1. 追踪 content 侧首次启动状态；
2. 追踪 background 请求与取消；
3. 审计现有测试缺口。
分别返回证据、候选根因和建议测试，主代理等待全部结果后统一判断。
```

主代理仍负责：

- 合并相互矛盾的结论；
- 确定单一状态所有者；
- 决定最终方案；
- 运行整体测试；
- 给出完整交付说明。

## 17. 权限、沙箱、网络和密钥安全

### 17.1 最小权限

- 默认权限足够时不要放宽；
- 只增加具体目录，而不是整个磁盘；
- 只授权任务所需域名；
- 发布、删除、外部消息和公开写入单独确认；
- 只读审计使用 Read Only；
- 不要因一次失败永久关闭沙箱。

桌面/CLI 可用 `/permissions` 查看或调整当前策略；CLI `/status` 可确认可写根目录和会话配置。

### 17.2 网络内容也是不可信输入

网页、Issue、README、模型输出可能包含提示注入或错误指令。处理方式：

- 只把它当数据和证据；
- 不执行网页要求的命令，除非与用户目标一致；
- 优先官方一手资料；
- 检查下载脚本和依赖；
- 不把本地密钥发送给网页或第三方工具。

### 17.3 密钥规则

- 不出现在 prompt、截图、日志、fixture、异常文本和 Git diff；
- 使用环境变量、操作系统凭据或本地 secret storage；
- 日志只记录 scope/是否存在，不记录值；
- API 错误正文可能回显请求信息，复制前脱敏；
- 轮换疑似泄露的密钥，不能只删除日志。

## 18. 成本、速度和上下文管理

### 18.1 用对模式和模型

- 简单搜索、格式化和小改动：快速/低推理；
- 状态机、并发、架构、安全：更高推理；
- 先用快速代理做只读定位，再让强推理代理做决策；
- 不要让多个昂贵代理重复读取相同大文件。

### 18.2 控制上下文

- 每个聊天围绕一个连贯目标；
- 新问题新聊天，相关 follow-up 留在原聊天；
- 长任务阶段性摘要；
- 重要结论写进文件，不依赖聊天永远保留；
- 使用 `/compact` 前确认关键约束已经在仓库或摘要中；
- 用 `/status` 检查上下文，而不是等模型明显遗忘后再处理。

### 18.3 控制无效迭代

当连续两三次补丁都没有解决问题：

1. 停止继续加条件；
2. 回到最后一个已知良好检查点；
3. 重画数据/状态流；
4. 增加能区分根因的观测；
5. 写最小失败测试；
6. 重新计划。

## 19. 常见失败模式

### 19.1 “修复所有问题”

问题：范围无限，成功不可定义，代理会边猜边扩大修改。
替代：列出优先级最高的可观察故障，每次修一个不变量。

### 19.2 聊天成为唯一知识库

问题：压缩、换任务或换代理后规则丢失。
替代：把长期规则写入 `AGENTS.md`，行为写入测试，过程写入脚本/CI。

### 19.3 看到例子就编码例子

问题：corner case 无穷无尽。
替代：从例子推导结构、时间、覆盖、所有权或协议不变量。

### 19.4 测试数量很多，但真实产品仍坏

问题：测试层级偏纯函数，缺少浏览器生命周期和 UI 证据。
替代：建立状态机、集成、smoke/E2E 和人工验收的分层证据。

### 19.5 一个文件/一个任务越来越大

问题：上下文难检索，review 困难，代理复制局部坏模式。
替代：按状态所有权拆模块；用架构检查限制大小；大目标分阶段提交。

### 19.6 为兼容而兼容

问题：每条旧协议都扩大状态空间和测试矩阵。
替代：只兼容当前真实、文档化且仍在支持的契约；过期路径明确移除。

### 19.7 把权限问题用 Full Access 解决

问题：省下一次审批，却扩大所有后续动作风险。
替代：定位需要的最小目录、域名、命令前缀或一次性批准。

### 19.8 Codex 说“已完成”就直接相信

问题：完成陈述不是证据。
替代：查看 diff、命令退出码、测试结果、截图/运行时日志和残余风险。

## 20. CaptionAI Duo 推荐操作流程

本仓库已经提供：

- 根指引：[../AGENTS.md](../AGENTS.md)
- 架构：[../ARCHITECTURE.md](../ARCHITECTURE.md)
- 可验证开发流程：[VERIFIED_DEVELOPMENT.md](VERIFIED_DEVELOPMENT.md)
- 模块指引：`content/AGENTS.md`、`background/AGENTS.md`、`popup/AGENTS.md`、`tests/AGENTS.md`
- 完整门禁：`npm run check`
- GitHub Linux/Windows CI：`.github/workflows/verify.yml`

### 20.1 新功能

1. 新聊天打开仓库；
2. `/plan`；
3. 使用本章任务模板定义验收标准；
4. 让 Codex 读根和目标目录 `AGENTS.md`；
5. 确认方案没有新状态所有者和不必要权限；
6. 允许实现、测试、文档更新；
7. 运行 `npm run check`；
8. `Ctrl + Shift + G` 打开 Review；
9. `/review` 审查未提交改动；
10. 完成相关 Chrome smoke test；
11. 建立单一目的提交。

### 20.2 日志驱动 Bug

推荐直接复制：

```text
先不要改代码。读取 AGENTS.md 和相关子目录指引。

现象：<用户看到什么>
时间：<视频位置/发生时刻>
日志：<最小脱敏事件窗口>
期望：<可观察结果>

请：
1. 画出从输入到显示的状态路径；
2. 区分调度、网络、模型、解析、提交、绘制问题；
3. 找到第一处错误状态和所属模块；
4. 给出一般性不变量，不得使用短语/视频/时间戳特判；
5. 设计一个修复前失败的最小测试；
6. 等我确认根因后再实施。
```

确认后：

```text
按确认的不变量实施。先加入失败测试，再修改状态所有者；运行最近测试和 npm run check，
审查是否有竞态、重复状态、兼容残留或密钥泄露。最后说明根因、通用性、证据和未运行的手测。
```

### 20.3 UI 修改

必须补充：

- 窗口固定宽度下的截图/视觉检查；
- 键盘和 focus 状态；
- 中英文文案；
- Chrome 扩展重载后的首次打开；
- `storage.local`/`sync` 数据边界；
- 一个用户动作只触发一次副作用；
- 全屏和非全屏（如果影响字幕覆盖层）。

### 20.4 网络/模型供应商修改

必须补充：

- 官方当前接口文档；
- 任意 SSE 字节切块；
- usage-only 事件；
- 非流式兼容响应（仅在当前契约要求时）；
- connect/body timeout 区分；
- 取消不是超时；
- Chromium 底层错误保真；
- API Key 不进入日志；
- 不因一个主机名特判破坏通用 OpenAI-compatible 路径。

## 21. 可复制模板

### 21.1 功能实现模板

```text
目标：

用户行为：
- 当前：
- 期望：

验收标准：
1.
2.
3.

必须保持：
-

非目标：
-

授权范围：
- 可修改：
- 不可执行：提交/推送/发布/外部消息/扩大权限（按需调整）

验证：
- Focused test：
- Full check：
- Manual smoke：

先探索和计划，确认状态所有者与风险后再实现。
```

### 21.2 根因分析模板

```text
只诊断，不修改。

现象：
证据：
期望：

请输出：
1. 完整状态/数据流；
2. 第一处异常状态；
3. 候选根因及支持/反对证据；
4. 区分候选根因的最小实验；
5. 被违反的一般不变量；
6. 修复范围与回归风险。
```

### 21.3 实施模板

```text
按照已确认计划实施。

- 先添加修复前会失败的行为测试；
- 在不变量所有者处修复；
- 不增加例子特判、重复状态或推测性兼容；
- 保留取消、超时和原始错误原因；
- 运行 focused test、npm run check、git diff --check；
- 最后 review 整个 diff，并报告未运行的浏览器检查与残余风险。
```

### 21.4 Review 模板

```text
只审查当前 diff，不修改。

按优先级检查：
1. 用户意图和验收标准是否满足；
2. 正确性、竞态、首次启动、取消和旧回调；
3. 数据验证、权限、密钥和日志；
4. 例子特判、不必要兼容、重复状态；
5. 测试是否证明一般性不变量；
6. 文档、加载顺序和 CI 是否同步。

只报告可操作问题，提供严重度、文件位置、触发条件、后果和最小修复方向。
如果没有问题，明确说明残余测试盲区。
```

### 21.5 重构模板

```text
目标是重构而不改变行为。

先建立/确认行为基线，列出模块边界和迁移顺序。每一步保持可运行、可测试、可回滚；
不要同时改变协议、产品默认值或 UI。为加载顺序和跨模块接口增加机械检查。
完成后比较重构前后测试、运行时入口、权限、存储 schema 和公开行为。
```
